# Temu Auto System V1 需求文档草稿

## 1. 背景

当前已有 `temu-auto-system-api.md` 记录了 Temu 商家后台多个业务动作的接口调用链，包括商品合规信息、商品实拍图、上新调价、商品库存、申报核价和产品加站。

V1 目标不是直接覆盖所有 Temu 后台能力，而是基于已取证接口，沉淀一个可控、可审计的自动化辅助系统，帮助运营人员减少重复点击、批量读取待办数据，并在人工确认后执行高风险提交动作。

## 2. 目标用户

- 运营人员：处理 Temu 商品合规、调价、库存、核价、加站等日常待办。
- 店铺负责人：查看自动化处理记录，确认高风险操作是否符合预期。
- 开发/维护人员：维护接口适配、风控字段获取、请求拦截与日志。

## 3. V1 范围

### 3.1 纳入范围

1. 商品合规信息
   - 查询商品当前合规详情。
   - 组装动态模板校验、编辑内容校验、最终提交所需参数。
   - 提交前展示变更摘要，由人工确认。

2. 商品实拍图
   - 查询实拍图详情。
   - 获取上传参数。
   - 发起上传前预校验。

3. 上新调价
   - 查询商品调价信息、优惠券最小金额、调价原因和灰度权限。
   - 支持按商品发起批量调价提交。

4. 商品列表库存
   - 读取商品表格行数据。
   - 打开库存修改数据。
   - 保存库存修改前进行人工确认。

5. 申报核价
   - 查询待确认申报价格。
   - 按接口返回 SKU 和价格生成提交请求。
   - 提交前展示价格差异和影响商品。

6. 产品加站
   - 快速筛选“商品信息待确认”列表。
   - 查询待确认商品详情。
   - 根据详情生成加站确认提交参数。

### 3.2 暂不纳入范围

- 自动绕过 Temu 风控或验证码。
- 保存完整 Cookie、Anti-Content、Authorization、anti-token 等可复用认证信息。
- 无人工确认的真实写操作批量提交。
- 未在接口文档中取证的页面能力。
- 多店铺权限体系和团队协作审批流。

## 4. 业务流程

### 4.1 通用流程

1. 用户选择业务模块和店铺上下文。
2. 系统读取当前页面或浏览器会话中的必要动态请求头。
3. 系统调用查询类接口获取待办数据。
4. 系统解析接口返回值，生成可读的待办清单。
5. 用户选择单个或多个待办项。
6. 系统根据接口文档组装写操作请求体。
7. 系统展示提交摘要，包括商品、SKU、价格、库存、合规任务、站点等关键变更。
8. 用户二次确认后，系统发起真实提交接口。
9. 系统记录操作结果、请求摘要、响应状态和失败原因。

### 4.2 写操作安全流程

所有真实写接口必须满足：

- 提交前有二次确认。
- 展示变更前后差异或提交影响范围。
- 不记录完整敏感请求头。
- 支持拦截/演练模式，用于只构造请求而不发出请求。
- 失败时保留可排查的非敏感日志。

## 5. 接口映射

| 模块 | 查询接口 | 写/提交接口 | 风险等级 |
| --- | --- | --- | --- |
| 商品合规信息 | `/compliance_property/query_detail`、`/query_dynamic_template`、`/check_edit_compliance` | `/compliance_property/edit_compliance` | 高 |
| 商品实拍图 | 实拍图详情、上传参数、预校验接口 | 待补充真实上传/提交链路 | 中 |
| 上新调价 | 调价信息、优惠券最小金额、调价原因、灰度权限查询 | `/goods/price/batch_update` 类批量调价接口 | 高 |
| 商品库存 | 商品表格行数据、库存弹窗数据 | 保存库存修改接口 | 高 |
| 申报核价 | `/price/bargain-no-bom/query` | `/price/bargain-no-bom/batch` | 高 |
| 产品加站 | `/robin/searchForChainSupplier`、`/queryOnlineGoodsForGoodsCommitConfirm` | `/goods/bindSiteConfirmForPrice` | 高 |

## 6. 核心数据规则

- `mallid` 必须来自当前店铺上下文，不允许跨店铺复用。
- `goodsId`、`spuId`、`productId`、`skuId`、`skcId` 必须来自当前查询结果。
- `siteVersion`、`priceConfirmKeyStr` 等版本/确认字段必须来自提交前最近一次详情查询。
- 价格字段按接口单位处理，文档中观察到的价格多为“分”。
- 商品合规任务以 `task_type` 作为业务类型标识，以 `task_id` 作为具体任务标识。
- 写操作请求体不得手工伪造关键业务字段，应由查询接口结果派生。

## 7. 验收标准

1. 用户可以在系统中看到五类业务模块入口：合规、实拍图、调价、库存、核价、加站。
2. 每个模块至少支持读取接口数据并展示待办摘要。
3. 高风险提交接口默认进入演练模式，不会直接发出真实请求。
4. 切换到真实提交前，系统必须展示二次确认弹窗和提交摘要。
5. 敏感请求头不会写入配置文件、日志或需求文档。
6. 同一条待办的提交参数能追溯到对应查询接口返回值。
7. 请求失败时，用户能看到模块、商品标识、接口名、错误码和非敏感错误信息。

## 8. 待确认问题

1. V1 是否只做本地自动化工具，还是需要多人使用的后台系统？
2. 系统是否运行在已登录的浏览器环境中，并复用浏览器会话发起请求？
3. 写接口是否默认全部只做“构造请求 + 拦截演练”，等人工确认后再逐步开放真实提交？
4. 是否需要支持多店铺切换？如果需要，店铺权限和操作日志归属如何定义？
5. 商品实拍图模块是否需要真的上传图片，还是 V1 只做到预校验与参数准备？
6. 调价、库存、核价是否需要批量处理上限，例如单次最多 10 条或 50 条？
7. 是否需要把每次真实提交结果导出为 CSV/Excel 供运营复盘？

## 9. 主要风险

- Temu 接口和风控字段可能变化，导致自动化请求失效。
- 写接口会影响商品状态、价格、库存或发布流程，必须避免误提交。
- 认证信息处理不当会带来账号安全风险。
- 依赖页面当前数据时，如果跨页复用 `goodsId`、`skuId`、`siteVersion`，可能提交到错误商品。
- 接口文档来自取证样本，部分异常分支和失败响应尚未覆盖。

## 10. 下一步设计建议

V1 进入设计阶段前，建议先确认：

- 产品形态：浏览器插件、本地桌面工具、脚本 CLI，还是 Web 管理台。
- 写操作策略：演练优先，还是允许特定模块真实提交。
- 首个落地模块：建议从“产品加站”或“申报核价”开始，因为接口链路相对完整，且文档中已经记录了拦截验证方式。
